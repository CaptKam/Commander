Follow The Money (FTM) — Complete System Outline
Overview
FTM is a multi-agent crypto intelligence platform that tracks Trump family crypto connections, correlates political events with market movements, and generates quantitative trading signals. Five autonomous systems communicate via deployed Replit URLs.

Architecture
POTUS (Intelligence) → Oracle (Prediction) → Jarvis (Risk Review) → Commander (Execution)
                              ↑
                        Atlas (Technical Analysis + Macro Data)

System 1: POTUS — Political Intelligence
Purpose: Real-time political event detection, entity tracking, knowledge graph
URL: https://potus-CaptKam.replit.app
Color: Red (#ef4444)
Tech: Express + PostgreSQL (local helium) + Claude AI + Drizzle ORM
6 Autonomous Agents:

News Sentinel (every 30 min) — breaking news, sentiment analysis, free crypto APIs + Claude web search
Policy Scanner (every 2 hours) — Federal Register, SEC, Congress.gov, CFTC
Whale Watcher (every 15 min) — blockchain wallet monitoring, WLF treasury, whale transactions
Statement Tracker (every 1 hour) — Trump family + crypto leader statements, evasion detection
Event Clusterer (every 4 hours) — groups related events, detects suspicious timing (insider signals)
Entity Linker (every 6 hours) — discovers knowledge graph connections automatically

Agent Pipeline: Gather raw data → Send to Claude AI → Parse JSON findings → Store in PostgreSQL → Forward to Oracle (significance ≥ 5) → Create alerts (significance ≥ 7)

notes (68 rows) — researcher annotations

System 2: Oracle — Quantitative Prediction Engine
Purpose: 10-model quant engine, generates BUY/SELL/HOLD predictions with Kelly Criterion sizing
URL: https://oracle-CaptKam.replit.app
Color: Amber (#f59e0b)
Tech: Express + PostgreSQL (local helium) + Drizzle ORM
Pipeline Flow:
Signal arrives → Save to signals table → Fetch POTUS context → 
Load price history → Run Bayesian model → Run Logistic model → 
Run momentum/zscore → Kelly Tournament ensemble → 
Generate prediction per asset → Apply Atlas macro regime multiplier → 
Save to predictions table → Forward top prediction to Jarvis
10 Quantitative Models:

Bayesian probability updating
EWMA momentum
Cosine similarity event matching
Kelly Criterion position sizing
Sharpe/Sortino/Calmar ratios
Logistic regression with gradient descent
Volatility-adjusted confidence
Z-score mean reversion
Exponential event decay weighting
Monte Carlo simulation

Kelly Betting Tournament: Based on arXiv:2602.09982, replaces hardcoded ensemble weights with academically grounded model evaluation
8 Target Assets: BTC, ETH, SPY, GLD, USO, DJT, RUM, LMT
Price Ingestion: FMP API every 15 minutes (each symbol individually)
Atlas Macro Integration: Before every prediction, Oracle calls Atlas GET /api/v1/macro-regime. Returns Fear & Greed index, stablecoin supply, regime classification (EXTREME_FEAR/FEAR/NEUTRAL/GREED/EXTREME_GREED), and Kelly sizing multipliers. Currently F&G = 23 (EXTREME_FEAR), long_size_multiplier = 1.25x.
Database Tables:

signals — incoming political events
predictions — generated BUY/SELL/HOLD with confidence, Kelly %, entry price
price_snapshots — FMP price history for 8 assets
model_weights — persisted Bayesian and logistic regression weights

Key Endpoints:

### System 5: Commander — Trade Execution (Separate Codebase)

**Purpose:** Paper trading via Alpaca API, harmonic pattern execution

**URL:** Runs on its own Replit (Pattern Bot)

**Color:** Platinum (#e2e8f0)

**Tech:** Express + React + Vite + Tailwind + PostgreSQL + Drizzle ORM + Alpaca API

**NOT connected to Jarvis yet.** Running independently as a harmonic pattern trading bot.

**Trading Engine:**
- 30-second scan loop with mutex lock (no overlapping scans)
- Scans 23 symbols (9 crypto + 14 equities) across 1D and 4H timeframes
- 5 enabled patterns: Gartley, Bat, Alt Bat, Butterfly, ABCD
- 2 disabled patterns: Crab, Deep Crab (low win rate)
- Alpaca paper trading (https://paper-api.alpaca.markets)
- Position sizing: 7% crypto, 5% equity per trade
- Maximum 10 open positions, maximum 20 daily trades
- TP1 at 38.2% AD retracement, TP2 at 61.8% AD retracement
- Stop loss: beyond X (retracement patterns) or beyond D (extension patterns) + 5% XA buffer

**Pipeline:**
```
Alpaca candle data → Pivot detection (5-bar left/right) → 
Fibonacci ratio matching (±5% tolerance) → 7-rule quality filter → 
Phase C screener → DB dedup → Zod validation → Save to PostgreSQL → 
Place Alpaca limit order → Exit manager tracks fills → 
TP1 (50% close) → TP2 (remaining close) → Software SL (market exit)
```

**7 Quality Rules:**
1. XB ratio 0.2–1.0
2. XD within pattern-specific bounds
3. AC ratio 0.2–1.0
4. Risk/reward ≥ 1.0
5. Profit target ≥ 2.0%
6. Fibonacci proximity ≤ 15% average deviation
7. Pattern age within timeframe window (1D: 14 days, 4H: 7 days)

**Exit Management:**
- Phase 1: Check pending entries for fills
- Phase 2: Check filled entries for TP order fills
- Phase 3: Software SL — monitor price vs stop loss (market exit if breached)
- Crypto monitor: WebSocket streaming prices for instant TP/SL detection
- Fallback: REST position data if no stream price

**Real-Time Price Streaming:**
- Crypto: wss://stream.data.alpaca.markets/v1beta3/crypto/us (24/7)
- Stocks: wss://stream.data.alpaca.markets/v2/sip (market hours only)
- Auto-reconnect with exponential backoff
- Heartbeat ping every 30s

**Alpaca Safety:**
- Anti-422 formatters: crypto qty max 9 decimals, stock qty whole shares
- Rate limiter: 200 req/min (free tier), warns at 80%
- Trading rate limiter: 60 req/min reserved for trading calls
- Hard-to-borrow retry: GTC → DAY fallback for stock shorts
- Buying power check before every order

**Current Dashboard (App.tsx):** 1,100 lines, 4 tabs (Terminal, Analytics, Risk Engine, Logs), 3-panel Terminal layout. Working API wiring but hard to read — redesign planned as single-page vertical layout.

Replit Quirks Documented

Deployed instances have separate DB context from dev instances
Egress proxy blocks Neon (*.neon.tech) on new projects
Each Replit has isolated PostgreSQL at helium:5432
Deployed "private" sites redirect API calls to login — must be public
Secrets with KEY=value format cause URL parse errors
tsx watch doesn't always hot-reload — use Stop/Run
BusyBox environment: no -x, -q, -E, -o flags for grep; head requires -n 3; cat doesn't support -n
Environment variables set via export don't propagate to child processes spawned by npm run dev
Drizzle push deletes tables/columns not in schema.ts — always use raw SQL for additive changes
